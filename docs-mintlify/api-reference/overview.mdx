# API Reference

QuickCommerce Tracker provides a comprehensive REST API for accessing pricing data, managing alerts, and integrating with your applications.

## Base URL

```
https://api.tracker.creatosaurus.in/v1
```

## Authentication

All API requests require authentication using bearer tokens:

```bash
Authorization: Bearer YOUR_API_KEY
```

### Getting an API Key

1. Go to **Settings** → **API & Integrations**
2. Click **Generate API Key**
3. Copy and store securely
4. Use in all API requests

## Request Format

### Headers

```http
GET /products HTTP/1.1
Host: api.tracker.creatosaurus.in
Authorization: Bearer YOUR_API_KEY
Content-Type: application/json
Accept: application/json
```

### Query Parameters

Include parameters in request URL:

```
GET /products?category=Fruits & Vegetables&limit=10&page=1
```

### Request Body

For POST/PUT requests, send JSON:

```json
{
  "sku": "SKU001",
  "trigger": "price_below",
  "targetPrice": 100
}
```

## Response Format

### Success Response

```json
{
  "status": "success",
  "data": {
    "products": [...]
  },
  "pagination": {
    "page": 1,
    "limit": 50,
    "total": 250
  }
}
```

### Error Response

```json
{
  "status": "error",
  "error": {
    "code": "INVALID_PARAMETER",
    "message": "Missing required parameter: category",
    "details": {
      "field": "category",
      "requiredValues": ["fruits", "bakery", "dairy"]
    }
  }
}
```

## HTTP Status Codes

| Code | Meaning |
|------|---------|
| 200 | Success |
| 201 | Created |
| 204 | No Content |
| 400 | Bad Request |
| 401 | Unauthorized |
| 403 | Forbidden |
| 404 | Not Found |
| 409 | Conflict |
| 429 | Rate Limited |
| 500 | Server Error |

## Rate Limiting

Check response headers for rate limit info:

```http
X-RateLimit-Limit: 600
X-RateLimit-Remaining: 599
X-RateLimit-Reset: 1678977600
```

Rate limits vary by plan:

| Plan | Req/min | Req/day |
|------|---------|---------|
| Free | 60 | 10,000 |
| Pro | 600 | 100,000 |
| Business | 1,200 | 500,000 |

## Pagination

For endpoints returning multiple results:

```json
{
  "data": [...],
  "pagination": {
    "page": 1,
    "limit": 50,
    "total": 250,
    "pages": 5
  }
}
```

Use `page` and `limit` parameters:

```
GET /products?page=2&limit=50
```

## Filtering

Most endpoints support filtering:

### Available Filters

- **category** - Product category
- **region** - Region/city code
- **platform** - Quick-commerce platform
- **dateFrom** - Start date (YYYY-MM-DD)
- **dateTo** - End date (YYYY-MM-DD)
- **minPrice** - Minimum price
- **maxPrice** - Maximum price
- **stock** - Stock status (in_stock, out_of_stock)

### Filter Examples

```
GET /products?category=fruits&stock=in_stock&maxPrice=100

GET /products/SKU001/history?dateFrom=2025-03-01&dateTo=2025-03-17
```

## Sorting

Sort results using `sort` parameter:

```
GET /products?sort=price_asc
GET /products?sort=price_desc
GET /products?sort=updated_desc
```

Available sorts:
- `price_asc` / `price_desc` - By price
- `updated_asc` / `updated_desc` - By last updated
- `popularity_asc` / `popularity_desc` - By tracked users
- `name_asc` / `name_desc` - By name

## Best Practices

✓ **Do:**
- Cache responses appropriately
- Use pagination for large datasets
- Implement exponential backoff for retries
- Monitor rate limit usage
- Log API requests for debugging

✗ **Don't:**
- Hardcode API keys
- Poll API unnecessarily
- Ignore error responses
- Make synchronous calls in UI
- Share API keys publicly
